# Subtree-INDEX template

This file is a **derived directory map** regenerated mechanically by `doc-integrity` at every close gate. **Do not edit by hand** — edits are overwritten on regeneration.

`doc-integrity` writes one of these in each indexed subtree:
- `docs/permanent/domain/capabilities/subtree-INDEX.md`
- `docs/permanent/domain/aggregates/subtree-INDEX.md`
- `docs/permanent/features/subtree-INDEX.md`
- `docs/permanent/features/design-specs/subtree-INDEX.md`
- `docs/permanent/flows/subtree-INDEX.md`
- `docs/permanent/decision-records/DR/subtree-INDEX.md`
- `docs/permanent/decision-records/ADR/subtree-INDEX.md`

## Format

```markdown
# Subtree index: <directory-name>

Regenerated by doc-integrity at <ISO>.
This file is a derived view. Do not edit by hand — edits are overwritten on regeneration.

| ID | Status | Name | Summary |
|---|---|---|---|
| <id> | <status> | <name> | <summary, ≤120 chars> |
| ... |
```

## Regeneration source

For each file in the subtree (excluding `subtree-INDEX.md` and `editorial-log.md`):
- `id` — the `id:` or `slug:` from frontmatter.
- `status` — the `status:` from frontmatter.
- `name` — the `name:` from frontmatter, or the first H1 if `name:` absent.
- `summary` — the `summary:` from frontmatter, or the first line of `## Description` truncated to 120 chars if `summary:` absent.

Files with no readable frontmatter are flagged in the integrity report, not included in the index.

Sorted by `id` ascending. All statuses included (proposed, accepted, superseded, deprecated, withdrawn) so the index is a complete map.

## Why this exists

A subtree may grow to dozens or hundreds of files over many phases. When the orchestrator or a human reviewer wants to know "what exists in this subtree?", the index lets them see in one screen rather than reading every file's frontmatter individually. The mechanical regeneration is the safeguard against staleness.

The orchestrator uses the index when constructing manifests for subagents that need cross-subtree context (e.g., when `domain-design` runs at phase scope and benefits from knowing what capabilities exist across the project's history). For manifest construction within an increment's declared scope, the orchestrator uses the increment-scope's named references directly — the index is for the cross-cutting cases.
